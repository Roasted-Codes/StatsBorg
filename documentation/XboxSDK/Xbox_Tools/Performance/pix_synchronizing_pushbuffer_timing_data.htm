Synchronizing Push-buffer and Timing Data

PIX supports capturing timing data without push-buffer data. It can also capture push-buffer data without timing data. It is at its most powerful, though, when both types of data are captured and work together to provide a more complete picture of what is happening in the game.

Since push-buffer capture takes so long, it is not possible to do it at the same time that timing data is being captured. Trying to capture both at the same time would skew the timing results so much that they would be useless. This means that when both types of data are requested, the data must be captured in two passes. This imposes a requirement on the game being analyzed. It must be able to repeat the rendering commands from the previous frame in the current frame. This ensures that the events in the push-buffer capture match up with the events in the timing capture.

If the game does not do this properly, PIX will detect that the timing and push-buffer events don't match, and it will then give the user an option to examine just the push-buffer data or just the timing data.

To determine whether it must repeat a frame, a title should call D3DPERF_QueryRepeatFrame. This function returns TRUE if PIX expects the title to repeat the previous frame. The title can call this function at the beginning of its frame to determine whether it should update the scene or re-render it from the previous frame. When PIX is capturing data, D3DPERF_QueryRepeatFrame will return TRUE after Present is called at the end of the timing capture frame, until Present is called at the end of the push-buffer capture frame.

The following code snippet demonstrates how to use D3DPERF_QueryRepeatFrame.

// Run the game loop, animating and rendering frames.
for( ;; )
{
   HandleInput();
   FrameMove();
   Render();
   
   // Render the frame again if the PIX tool is running and has 
   // requested us to repeat the frame. Note that we don't move, 
   // but rather re-render the scene exactly as before.
   if( TRUE == D3DPERF_QueryRepeatFrame() )
   {
      Render();
   }
}

Note    Many games provide a separate means of disabling scene updates so that they can "freeze" the game at a particular point. If the game doesn't wish to support scene repeating on-demand, a user can just put the game in "freeze" mode and then capture timing and push-buffer data.

It may be possible for PIX to do more than the behavior previously described if the timing events and the push-buffer events don't match. It might make sense for PIX to analyze the two event lists and discard events that don't have a match. The remaining events might still provide some useful information. This will require some experimentation to determine if it is worthwhile.
