Marketplace Sample

The Marketplace sample demonstrates proximity-based conversations for large numbers of users over speaker and headset. It shows a method for dynamically allocating bandwidth using simple peer prediction for a scalable mix of peer-to-peer and player-server voice traffic.

Overview

This program is more of a demo than a sample. It demonstrates the concepts discussed in the voice partitioning white paper. The technical areas this sample covers are:

  • Speaker and headset output through XHV
  • Coalescing voice packets to save bandwidth
  • Proximity-based voice partitioning with low server overhead

Note    This sample does not comply with logon, and other Xbox Live TCRs, because it demonstrates a way to do voice partitioning rather than implementing a full game.

The Challenge of Voice Partitioning

Partitioning voice can be costly in terms of both bandwidth and CPU time. As a result, you usually need to partition the space over which voices can be heard. If a player is inside the partition space of another player, they can hear each other speak. Otherwise, they cannot.

Voice partitioning can be problematic. For instance, you can partition the space based solely on the n closest players to any individual. If you have a player standing in the middle of a circle of people, and several other players in the circle are talking, the person in the center will have his bandwidth saturated with voice packets within the partition space of all others in the circle.

There are several possible solutions to this problem, each with its own advantages and disadvantages. You could have the server calculate all of the player-to-player routes. This costs in bandwidth and CPU cycles, however, and introduces significant delays if players talk while the routes are recalculated. On the other hand, you can precalculate routes. This means that players can hear only the closest n people. Although in this case, players are unable to hear someone who isn't in the closest n people to them, even if none of those people are talking.

A Proposed Solution

The solution demonstrated in Marketplace involves a single bit per player being sent on the update packet from the server. This bit specifies whether a given player is talking or not. The player-to-player voice routes are completely calculated on each player based on the talking bits, so oversaturating a player can be avoided. Even if the update comes a little late, oversaturating a player will not last more than a single network update (50 ms by default), which allows adequate time to recover.

A priority is determined for each proximal player-to-player pair, and then bandwidth is allocated round-robin, with a player being removed from the send or receive set as soon as its bandwidth quota is used up. This quota could easily be scaled based on players' QoS bandwidth numbers, so some players would automatically receive or send more data than others based on what their connection allows.

Priority is given to players near a given player, players that aren't moving (and thus are likely to be in an established conversation), and players facing other players. Additional priority modifiers could be implemented, including higher priority for friends.

Remarks

Because of the flexibility in network architecture required for this sample (for example, support for both peer-to-peer and full server-forward architectures), there may not be much code in it that you can use directly in your own programs. However, the program is provided as a sample, rather than a demo, in the hope that you find the code useful.

Also, because of the added complexity in making a visual demonstration (which includes bots, sorted sprites, a soundbit editor for the bots, and so forth), there is a lot of code here that is not directly applicable to voice partitioning. The full voice path code is spread throughout the sample. Most of the partitioning code is in VoiceMgr.cpp, but some other important concepts, such as 3D sound buffer caching, are implemented in Audio.cpp.

The network stats (accessed by pressing the white button) shows only the traffic used for voice. If voice is piggybacked on a data packet, the 44 bytes for the packet header are not counted. If the voice is sent by itself (as in peer-to-peer), the 44 bytes are counted.

In peer-to-peer mode, the host bandwidth shows only the stats for voice going through the server; there would be no players locally in a massively-multiplayer game. The stats are given to show the impact on server bandwidth for using this method of voice partitioning.
