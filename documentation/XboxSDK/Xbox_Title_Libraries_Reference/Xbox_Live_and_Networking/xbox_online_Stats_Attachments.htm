Game Clips

Introduction

Many game titles benefit from enabling users to see not only the rankings of top players but also how the top players performed. These games could benefit further by enabling users to download and view content related to the top players' standings, content such as "ghost races" for the top 10 fastest laps, replays of the fastest time to complete a level, the most kills with a single explosion, and so on. Given this capability, a game could then upload game-created content in the form of attachments to game statistics. A game uploads these "Game Clips" from the local Xbox to the Xbox Live service for download by anyone playing the game.

Goals

Game Clips has the following goals:

  • Provide a secure process for allowing games to attach game-created content to stats rankings.
  • Allow users to find the attached game-created content and download it to their local Xbox.

Non-Goals

Game Clips is not intended to

  • Provide a process by which publishers can send content directly to us for distribution. Instead, content download should be used for this.
  • Allow titles to upload content to non–Xbox Live servers.
  • Allow generalized content upload.
  • Charge users for uploads. The title, not the player, must consume the cost.

Scenarios

The scenarios encompassed by Game Clips are limited only by the imagination of the game developer. In general, these scenarios relate to excellent game play that can be ranked in some way. The following scenarios detail a few possible game implementations of this feature. Keep in mind that this document covers only content upload and download. Developers must choose how to implement their game's client-side experience.

Ghost Race

Jim is playing Race Game on Xbox Live and completes the Tokyo race in first place. The game tells him he has completed the race fast enough to be ranked number 7 on Xbox Live and asks if he would like to upload the race he just completed so others can see it. He chooses to upload the race. He then looks at the stats for the Tokyo track and sees that he can download the top 10 races. He downloads the race from the number 1 ranked player and practices racing against it offline.

Replay

Jim has been unable to get past the bridge on the second level of Action Game so he decides to go to Xbox Live to get some help. He goes logs on and gets the list of players that have completed the level the fastest. When he sees the top 10 players' times, he can't believe anyone could beat the level that quickly, so he decides to see how some of the people completed the level. He downloads the replay of the #1 player and watches how the player gets past the bridge.

Hall of Fame

Jim sees an entry for "Hall of Fame" in Action Game and checks it out. Once there, he sees listings such as "Largest Explosion" and "Most kills with a single shot". He selects the "Largest Explosion" item and sees a list of the top 10 explosions created by users while playing the game. He selects the #1 entry, downloads a replay of the explosion, and watches it locally.

Design

Leaderboard Configuration A publisher can allow stats attachments on any of a title's leaderboards. The Stats service currently limits each title to 20 leaderboards. Publishers are allowed 1GB of attachment storage space per title. The total per-title storage quota of 1GB is enforced at leaderboard configuration time; the sum of the maximum consumable storage space for all of the title's leaderboards must not exceed 1GB. The maximum consumable storage space is a product of the maximum upload size and the number of slots on that leaderboard for which upload is allowed. The maximum upload size for a single attachment is 2MB.

Note    Leaderboards are also referred to as "scoreboards."

Publisher Submission

Within these limits, titles can manage per-leaderboard allotment, including maximum upload size and number of uploads per leaderboard. To take advantage of stats attachments for any of the title's leaderboards, games must use the Stats Configuration tool (see Tools) to submit a Stats Configuration package for each leaderboard to be attachment-enabled. This package contains:

  • LeaderboardID
  • Description (friendly name)
  • Number of uploads per leaderboard
  • Maximum upload size per leaderboard

Reconfiguration

The publisher can change the number of uploads per leaderboard after the game is released. The process is similar to the initial leaderboard configuration process described previously.

If a game reduces the number of uploads allowed on a given leaderboard, then the attachments for the ranks that drop below the upload threshold are deleted once the threshold is changed. For example, suppose the game initially keeps 20 packages for the top 20 races. If the title reduces the number of packages to only the top 5 races, we delete the other 15 packages at the time the upload threshold is changed.

Attachment Upload Verification

Games must carry out the stats-write and get an upload confirmation before uploading content. We are extending the stats-write functionality to communicate upload confirmation. For more information, see Proposed Function Call Flow for Stats with Attachment API later in this topic.

To prevent duplicate uploads, the Stats service allows only one attachment per stats write, even if the write request contains multiple user/attribute pairs. Titles can work around this restriction if necessary by separating user/attribute pairs into multiple write requests.

Stats-Write Functionality Enhancements

When the game writes the user's stats, the Stats service returns a value indicating whether the user has qualified for an upload. This is an enhancement to the stats-write functionality. Stats service includes new information in the stats-write response. Both the wire structure and Client APIs/Structures change accordingly.

  • Upload Confirmation A value that indicates whether there is an uploaded file associated with this user on this leaderboard.
  • Remote File Reference Remote file reference for the attachment.
  • Max Attachment Size Size of the largest upload package allowed. The client and the server (in case the client is hacked) must fail uploading any package above that size.
  • Number of Qualified Upload Slots Configured number of slots on the leaderboard for which uploads are allowed, counting from the top-ranked slot on the leaderboard.

    • Attachment Upload

    If the service confirms the attachment upload, the game calls the Upload API to store the user attachment on the server. This attachment content stays on the server indefinitely, with two exceptions:

    1. If the rank associated with an attachment drops below the per-leaderboard upload threshold, the content is no longer downloadable.
    2. If the title reconfigures the upload settings for the leaderboard, the threshold moves up to exclude the attachment and the content again becomes unavailable for download. For example, if the Most Kills leaderboard initially allows uploads for the top 10 ranked players and the threshold changes to 5, the uploads for players 6– 10 can no longer be seen.

    Titles must ask users whether to proceed with an attachment upload, if they qualify for it. For privacy, bandwidth cost, or time reasons, users may decide not to upload an attachment. Titles must make it clear that an upload is a now-or-never opportunity. Titles must not allow a user to take other actions before making the upload decision. For large attachments, the upload can take a long time. Per TCR, titles must provide user feedback for uploads longer than three seconds.

    Region and Version Conflict Management

    Xbox Live services do not have built-in ability to manage multiple title versions. Developers must ensure that all versions for a given TitleID are backward-compatible across auto-updates, and if the game ships in multiple regions under the same TitleID, developers must address any internationalization and localization issues. Xbox Live services can support publishers by resetting leaderboards and deleting old stats attachments if a title makes a change that breaks ross-version compatibility.

    Maximum Upload Size Enforcement

    The maximum size the server allows is returned to the client in the stats-write response. The client enforces the maximum size on every upload and fails all attempts to upload a package that is larger than the maximum size specified.

    Attachment Data Format

    Only the title determines the data format of the attachment. The stats service is passed a blob of data, it stores the blob, and passes the blob back. Although multiple files may be wrapped into a single file for upload purposes, it is always a single blob of data that is eventually uploaded to the server.

    Upload Errors

    In general, there is no guarantee that any upload is going to succeed, regardless of how immediate the upload attempt is. The likelihood of success does decrease, however, if the game delays the upload operation. The title should automatically retry failed attempts to upload an attachment. The title should tell the user that upload has failed and ask the user whether to try again. The title can call the Upload API again, but user confirmation is required.

    Upload API errors include the following cases:

    • Server is too busy. Keep trying.
    • Upload is interrupted. "Resumable" uploads are not supported. Asking the user for permission to try again is the only recourse.

    Note    Uploading empty attachments is not supported and will result in an error.

    Destructive Uploads

    Posting a new rating that qualifies for the attachment upload deletes any attachment that the user posting a rating might have had previously. If the user chooses not to upload (or the upload fails) neither the old nor the new attachment is available.

    If a ranking drops below the upload threshold, the attachments associated with the below-threshold ranks are deleted. If that user's same rank moves above the upload threshold, there will be no attachment associated with it. If a user's rank drops below the upload threshold during upload, the upload will continue as normal. If the ranking is below the threshold after the upload is complete, the stats service drops the content at that point.

    Metadata

    For internal auditing and tracking purposes, the Xbox Live service stores a minimum amount of metadata for each attachment uploaded to the service. This data includes:

    • Gamertag and PUID of user uploading attachment.
    • Signature assigned by the signature server.
    • Date and time the attachment was uploaded.
    • Machine PUID of Xbox the attachment was uploaded from.
    • Installed disc size of the attachment in blocks.
    • Compressed size of uploaded attachment.

    Attachment Download

    Uploaded attachments are available for download and viewing by any authenticated client running that title. Users discover available attachments by browsing through enabled leaderboards. Users can initiate the download process by choosing to download any of the available attachments.

    Download API

    The Download API for stats attachments provides the following information:

    • Remote File Reference (IN)
    • Uncompressed Size (OUT)
    • Compressed Size (OUT)
    • Attachment Content (OUT)

    Caching

    Client implementation of the Download and Upload API prevents repeated downloads if the content is already available on the hard disk. Client APIs store the downloaded content on the T: drive. The client statistics API enforces a limit of 20 cached attachments per title across all leaderboards.

    Tools

    A future XDK will provide a GUI "Stats Configuration" tool that enables publishers to configure leaderboards to work with Game Clips.

    Function Call Flow for Game Clips API

    Writing Scores

    1. As the user races, create a file of the replay and sign it with the Xbox Live key.
    2. When the user finishes the race, write the score to the server, as follows:
      1. Call XOnlineStatWrite. Store any game-specific information about the content as stat attributes, before initiating upload. Here are some examples of the type of information to store:
        • Game region, if the attachment content does not work across regions in that game.
        • Title version, to allow the game to block obsolete content after an auto-update.
        • Length of replay. If this is a high-score movie, a user may want to know how long it takes to watch.
        • The size of the file (compressed and uncompressed) is stored automatically by the API.
      2. Pump write task handle with XOnlineTaskContinue.
      3. If you get XONLINE_S_CAN_UPLOAD then
        1. Call XOnlineStatWriteGetResult to get a remote file reference.
        2. Close the task.
        3. Ask the user if he or she wants to upload, or alternatively, begin the upload automatically.
        4. Upload the attachment file, as follows:
          1. Call XOnlineStorageUpload with the remote file reference retrieved by the call to XOnlineStatWriteGetResult.
          2. Pump the upload with XOnlineTaskContinue.
          3. Optionally call XOnlineStorageGetProgress to find out how the upload is going and to display a progress bar.
          4. When the upload is complete, XOnlineStorageGetProgress returns S_OK. But there is still no guarantee the content is on the specified leaderboard. The user could have lost his or her standing on the leaderboard during the upload. For example, at the time the upload started, Mary qualified for 9th place; by the time the upload completed, she had fallen to 15th place, but the leaderboard isn't configured to store attachments for that low a ranking.

    Reading Scores

    Read scores and access downloads, as follows:

    1. The title creates XONLINE_STAT_SPEC structures that can contain XONLINE_STATs with any of three new reserved IDs:
      • The remote file reference of the download
      • The size uncompressed
      • The size compressed
    2. Call XOnlineStatRead or XOnlineStatLeaderEnumerate.
    3. Pump the task to completion.
    4. Call XOnlineStatReadGetResult or XOnlineStatLeaderEnumerateGetResults:

      • The title must pass in a buffer large enough for all the remote file references requested. The title can calculate this size as follows:

        (Number of users * (Size of remote file reference + Size of nickname)

      • The title can display information stored in the stats server about the download. For example, is there a download for this item? What is its size (compressed, uncompressed)? Is there any other title-specific information?
    5. Typically, the title displays an icon indicating that the user can download a replay for a particular statistic.
    6. If the user chooses to download the replay
      1. Call XOnlineStorageDownload with the remote file reference. (This does the download and any associated installing.)
      2. Pump the task to completion.
      3. Optionally call XOnlineStorageGetProgress to find out how the upload is going and to display a progress bar.

    Accessing and Managing Content on the Hard Disk

    • The attachment is automatically stored in TDATA by the API.
    • There will be APIs to access the attachment.
    • As the title downloads additional attachments, the API automatically deletes old saved replays.
    • The title can store these permanently by letting the user copy them to UDATA as saved games.
