XOnlineLogon

Creates an asynchronous task to log the specified users onto Xbox Live and to acquire access to the specified online services. 

HRESULT XOnlineLogon(
    const XONLINE_USER *pUsers,
    const DWORD *pdwServiceIDs,
    DWORD dwServices,
    HANDLE hWorkEvent,
    PXONLINETASK_HANDLE pHandle
);

Parameters

pUsers
[in] Array of XONLINE_USER structures that contain information about the players using the console who would like to participate in the online game session. The array must contain XONLINE_MAX_LOGON_USERS structures.

The order of users represented in the array is significant and must map one-to- one to the controller index (zero-based) of the game port that the player is using. If no player is using a specific game port, the XONLINE_USER structure at that index should be set to all zeroes.

Use the XOnlineGetUsers function to obtain XONLINE_USER structures that contain user information stored on the local Xbox hard disk or in Memory Units.

pdwServiceIDs
[in] Array of DWORD values specifying the services for which tickets will be obtained to enable future secure communication with Xbox Live services. Can be one or more of the following values:
Value Description
XONLINE_ARBITRATION_SERVICE Accesses the Arbitration service.
XONLINE_BILLING_OFFERING_SERVICE Accesses the Billing service.
XONLINE_CONTENT_AVAILABLE_SERVICE Accesses the content availability checking service. Allows use of the XOnlineOfferingIsNewContentAvailable function.
XONLINE_FEEDBACK_SERVICE Accesses the Feedback service.
XONLINE_MATCHMAKING_SERVICE Accesses the Matchmaking service.
XONLINE_MESSAGING_SERVICE Accesses the Messaging service.
XONLINE_NAT_TYPE_DETECTION_SERVICE Accesses the NAT Type Detection service.
XONLINE_QUERY_SERVICE Accesses the Query service.
XONLINE_SIGNATURE_SERVICE Accesses the Signature service. XOnlineLogon contacts the Signature Service and obtains a session signature key. The key is obtained before the logon task returns XONLINE_S_LOGON_CONNECTION_ESTABLISHED.
XONLINE_STATISTICS_SERVICE Accesses the Statistics service.
XONLINE_STORAGE_SERVICE Accesses the Storage service. You must specify this service in order to use the XOnlineStorageUpload and XOnlineStorageDownload functions.
XONLINE_STRING_SERVICE Accesses the String service.
XONLINE_TEAM_SERVICE Accesses the Team service.
dwServices
[in] Number of services in the pdwServiceIDs array.
hWorkEvent
[in, optional] Handle to an event that will be signaled when there is work to be done on the logon task. The event will be left unsignaled if the task is currently blocked by an asynchronous operation. This parameter can be NULL if no such event is required.
pHandle
[out] Pointer to a variable of type XONLINETASK_HANDLE that receives a task handle representing this logon task. The handle can then be used in calls to XOnlineTaskContinue to perform work associated with this task. This task must remain active while the players are signed in to the online service. This task processes the periodic online "heartbeat," which maintains the console's connection to the online service.

Return Values

The return value indicates whether the asynchronous task was successfully created:

Value Description
S_OK The asynchronous logon task was successfully created.
XONLINE_E_LOGON_NO_NETWORK_CONNECTION No network connection was detected.
XONLINE_E_OUT_OF_MEMORY There was not enough memory to create the asynchronous logon task.
XONLINE_E_TASK_BUSY Another logon task is currently being performed, or the previous logon task has not been closed. (In debug builds, an assert will occur when this value is returned.)

The return value does not represent the success or failure of the actual logon task. Once the logon task has been successfully created, XOnlineTaskContinue should be called until it returns a value other than XONLINETASK_S_RUNNING. At that point, XOnlineLogonTaskGetResults is called to determine the results of the logon. See XOnlineLogonTaskGetResults for a list of the possible return values.

After a successful logon, XOnlineGetLogonUsers and XOnlineGetServiceInfo must be called to determine the individual status for each user and service specified. See the Remarks section below.

Remarks

When specifying services with the pdwServiceIDs parameter, the title must explicitly reference the XONLINE_STATISTICS_SERVICE and XONLINE_STORAGE_SERVICE symbols in order to use these services. It does not suffice to specify the numeric values that correspond to these symbols.

XOnlineLogon returns an asynchronous task which must be pumped with XOnlineTaskContinue until that function returns a value other than XONLINETASK_S_RUNNING. A value other than XONLINETASK_S_RUNNING indicates that: 1) the logon has completed, 2) the logon has been aborted, or 3) an error has occurred. The results of the logon can then be obtained with a call to XOnlineLogonTaskGetResults.

A successful logon return code of XONLINE_S_LOGON_CONNECTION_ESTABLISHED does not necessarily indicate that all of the specified users and services were logged on. The results of the individual user logons must be determined by calling XOnlineGetLogonUsers. The results of the service logons are obtained with calls to XOnlineGetServiceInfo.

Once the logon is successful, XOnlineTaskContinue should continue to be periodically called for the logon task. As long as the online connection is valid, the return value will continue to be XONLINE_S_LOGON_CONNECTION_ESTABLISHED. If an error is returned, the logon task handle should be closed. If desired, another logon attempt can be made.

The logon process involves: 1) communicating with the authentication server to authenticate users using Kerberos, and 2) obtaining the tickets necessary for future communication with the requested online services. The service connections obtained by the logon process allow later calling online functions that use those services. XOnlineLogon can be called again, but all service connections obtained from the previous logon call will be lost. Therefore, it is recommended that titles first collect all user information and service IDs ahead of time, and then make a single logon call.

XOnlineLogon is not thread safe, so titles should avoid making a second XOnlineLogon call before the first logon task is completed. Additionally, before a new logon is attempted, the previous logon task handle must be closed with XOnlineTaskClose.
