Minimizing Private Pool Allocations

Here are some general tips to help minimize the use of private pool memory:

  • Call recv functions frequently. If this is done, the amount of pool memory used by each socket's recv buffer should be relatively small (probably only a couple of buffered packets per socket).
  • Keep track of the total amount of Xbox network subsystem private pool memory that could be allocated at any time. This is primarily based on the total number of sockets and the sum of their receive buffers. This includes sockets that are directly created by Winsock calls in the game, as well as sockets that are created as a result of DirectPlay and online function calls.
  • Consider running a debug build occasionally to catch these conditions. In a debug build, over-allocation of private pool memory will generate debugger warnings.
  • If necessary, increase the amount of Xbox network subsystem private pool memory via the cfgPrivatePoolSizeInPages member of XNetStartupParams in the XNetStartup call.
  • If necessary, consider decreasing the size of certain directly allocated sockets via the cfgSockDefaultRecvBufsizeInK member of XNetStartupParams in the XNetStartup call or via a setsockopt call with SOL_SOCKET and SO_RECVBUF on specific sockets. This will lower the maximum pool memory usage of each socket.
