Friends Overview

This documentation provides an overview of the Xbox Live Friends API. The overview covers the functions that make up the API, the purpose of each function, and how the functions should be called. The Friends sample in the XDK demonstrates a basic Friends list, and implements some of the functionality behind invitations and responses.

Displaying the Friends List

Xbox Live titles must enumerate and display the user's Friends list. There is some freedom in how each title can implement the user interface for the list. The general functionality requires some form of list display, sorted by the title's (or user's) choosing, of Friends members and their current state information.

Typically, Xbox Live titles allow the user to access the Friends list prior to the start of a game session, and also while a game session is paused. Titles do not need to enumerate the Friends list during gameplay.

Titles use the functions XOnlineFriendsEnumerate and XOnlineFriendsGetLatest to retrieve the Friends list and friend status information from the Friends service.

Function Description Memory Use
XOnlineFriendsEnumerate Creates task to enumerate Friends list. Yes - allocates the task, an internal copy of the Friends list, and buffers for getting updates from the service. In addition, allocates memory to facilitate looking up title names.
XOnlineFriendsGetLatest Retrieves Friends list when enumeration is complete. No - Copies the internal list to the buffer provided.

Note    The XOnlineFriendsStartup function must be called before calling XOnlineFriendsEnumerate.

To display the Friends list, first call XOnlineFriendsEnumerate. The XOnlineFriendsEnumerate function returns a task handle that the Title should pump with XOnlineTaskContinue until XOnlineTaskContinue returns XONLINETASK_S_RESULTS_AVAIL. At that point, an initial Friends listing is ready to be retrieved and displayed. To retrieve the Friends list, call the XOnlineFriendsGetLatest function.

The XOnlineFriendsGetLatest function takes as a parameter the address of a buffer that receives an array of XONLINE_FRIEND structures. This array represents the Friends list. The buffer should be large enough to contain MAX_FRIENDS number of XONLINE_FRIEND structures. The MAX_FRIENDS constant is defined to be a value of 100 in XOnline.h. The Friends list is sorted in the following order:

  1. Received game invitations.
  2. Sent game invitations.
  3. Received Friend requests.
  4. Sent Friend requests.
  5. Friends who are online and playing the same title.
  6. Friends who are online and playing a different title.
  7. Friends who are offline.

Within each of the preceding categories, the entries are sorted alphabetically by gamertag.

After displaying the initial Friends listing, the title should continue to call XOnlineTaskContinue with the handle returned by XOnlineFriendsEnumerate. Continuing to pump the handle causes the Friends service to process updates to the Friends list that it receives from other Xbox consoles connected to the Xbox Live service. To refresh the displayed Friends listing with these changes, the title should retrieve the updated list with another call to XOnlineFriendsGetLatest.

If the title is no longer displaying the Friends list (because game play has started, for example), the title should call XOnlineTaskClose on the task handle returned by XOnlineFriendsEnumerate.

Identifying Titles

The XOnlineFriendsEnumerate task uses the XOnlineFriendsGetTitleName function to maintain an in-memory cache of title names that can be used to display the names of the titles a friend is playing. XOnlineFriendsGetTitleName resolves title names by checking the memory cache first, checking a cache on disk second, and checking the service (by means of the XOnlineFriendsEnumerate task) third. If the title name is not immediately available, an empty string is returned instead. When a title name is resolved and put into the memory cache, the XOnlineFriendsEnumerate task returns XONLINETASK_S_RESULTS_AVAIL as it would for any other list change.

Titles can use the XOnlineTitleIdIsSameTitle function to determine whether a title from the Friends list is the same as their own. Titles can also use XOnlineTitleIdIsSamePublisher to determine whether the title is from the same publisher.

Titles must not identify titles in the Friends list by directly comparing title IDs against constants, or by performing string comparisons on the output of XOnlineFriendsGetTitleName.

XOnlineFriendsStartup: Request and Invitation Processing

Before calling any of the functions that manage Friends requests, removals, or game invitations, the title must create an asynchronous online task using the XOnlineFriendsStartup function. The only exception is XOnlineFriendsGetAcceptedGameInvite, which does not depend on the task created by XOnlineFriendsStartup. Calling XOnlineTaskContinue with this task causes work to be performed for the request and invitation functions. This task does not complete on its own. The title should continue to call XOnlineTaskContinue with this task (typically once per frame) until either the title no longer needs Friends functionality, or until XOnlineTaskContinue returns an error code indicating that an error has occurred with this task. If the title does not call XOnlineTaskContinue frequently enough, the XOnlineFriends functions may return the XONLINE_E_NOTIFICATION_TOO_MANY_REQUESTS error code. For more information about the asynchronous online task architecture, see the reference page for XOnlineTaskContinue.

The task returned by XOnlineFriendsEnumerate does not have to be active in order to process requests and invitations.

Adding and Removing Friends

Xbox Live titles must allow players to add and remove users from the Friends list. Membership on the Friends list is mutual: if User A is on User B's Friends list, then User B must also be on User A's Friends list. Also, the maximum number of users on the Friends list is limited to the value of the MAX_FRIENDS constant, defined in XOnline.h. As a result of these two constraints, a user whose Friends list is full is not able to add new users, and is also not able to be added to other user's Friends lists.

Use the following functions when adding or removing user's from the Friends list.

Function Description Memory Use
XOnlineFriendsRequest Requests that a user be added to a Friends list. No.
XOnlineFriendsAnswerRequest Responds to a request generated by XOnlineFriendsRequest. Yes - allocates memory for a request to the service.
XOnlineFriendsRemove Removes a user from the Friends list. Yes - allocates memory for an XONLINE_FRIEND structure.

Adding Friends: Sending and Receiving Friend Requests

Xbox Live titles use the XOnlineFriendsRequest function to request that a new user be added to the current user's Friends list. After calling XOnlineFriendsRequest, the title should call XOnlineTaskContinue with the handle returned by XOnlineFriendsStartup in order to process the request.

The user that is the recipient of the request receives a notification of type XONLINE_NOTIFICATION_TYPE_FRIEND_REQUEST in his or her notification queue. The title running on the Xbox console of the receiving user should periodically call the XOnlineGetNotification function for each of the users on that Xbox console to check for the receipt of such notifications. The title should then inform the user that he or she has a notification pending.

To determine the identity of the user who sent the request, the receiving title enumerates the Friends list for the receiving user. The requesting user is present in the receiving user's Friends list with a Friend state of XONLINE_FRIENDSTATE_FLAG_RECEIVEDREQUEST. (The receiving user is also present in the requesting user's Friends list with a Friend state of XONLINE_FRIENDSTATE_FLAG_SENTREQUEST.) The Friend state is stored in the Friendstate field of the XONLINE_FRIEND structure.

The title on the receiving user's Xbox console should call XOnlineFriendsAnswerRequest with an appropriate value from XONLINE_REQUEST_ANSWER_TYPE. To process the response to the request, after calling XOnlineFriendsAnswerRequest, the title should call XOnlineTaskContinue with the handle returned by XOnlineFriendsStartup.

If the receiving user accepts the request to be added to the Friends list, the title should respond with XONLINE_REQUEST_YES. If the user declines the request, the response should be XONLINE_REQUEST_NO. The receiving user also has the option to specify that he or she no longer receive Friends requests from the requesting user. In this case, the title should use the XONLINE_REQUEST_BLOCK value. If the receiving user accepts the request, the requesting user is sent a notification of XONLINE_NOTIFICATION_TYPE_FRIENDSTATUS. If the receiving user declines the request or blocks further requests, the requesting user is not sent any notification. Instead, the two users simply disappear from each other's Friends lists.

If the recipient of a Friend request blocks the requesting user (XONLINE_REQUEST_BLOCK), and the recipient subsequently sends a Friend request to that blocked user, the blocked user becomes unblocked. The blocked user is unblocked regardless of his or her response to the Friend request.

User Interface Considerations

A straightforward way for titles to provide the functionality for adding new friends is to allow a player to add other players from the current game session. In this scenario, the title already has access to the information needed to add the player to the Friends list. If the player is on the local Xbox console, the title can obtain the information through the XOnlineGetLogonUsers function. If the player is on a remote Xbox console, the information should be available from the Xbox console that is hosting the online-game session. For more information on network communication between Xbox consoles, see Xbox Secure Network Library.

The Xbox Dashboard screen for the Friends list supports a virtual keyboard which can be used to add friends based on user name. The Xbox Dashboard can be considered the default means for adding friends to the Friends list.

Removing Friends

Use the XOnlineFriendsRemove function to remove a user from the Friends list. The title should provide an opportunity for the user who requests the removal to confirm that he or she wants to proceed with the operation. Because Friends list membership is mutual, the user that requests the removal is also removed from the Friends list of the target user. The removal results in the target user receiving a XONLINE_NOTIFICATION_TYPE_FRIENDREQUEST notification. However, the title should not display this notification to the user. Each user should silently disappear from the other's list.

As with adding friends, the Xbox Dashboard screen for the Friends list supports removing friends as well.

Inviting Friends to Play a Game

Xbox Live titles must allow a player to invite users from his or her Friends list to participate in an online-game session. Such invitations are one method users can find out about available online-game sessions. Another method by means of Matchmaking.

Use the following functions to manage invitations to users on the Friends list.

Function Description Memory Use
XOnlineFriendsGameInvite Invite members of the Friends list to join a game. Yes - allocates memory for a request to the service.
XOnlineFriendsAnswerGameInvite Responds to an invitation generated by XOnlineFriendsGameInvite. No
XOnlineFriendsRevokeGameInvite Revokes a previously-generated game invitation. No
XOnlineFriendsGetAcceptedGameInvite Creates a task to retrieve a game invitation stored locally on the Xbox console. No. Also, this function does not require that the XOnlineFriendsStartup task be active.

Sending Invitations

Xbox Live titles use the XOnlineFriendsGameInvite function to invite a user on the Friends list to participate in an online-game session. In order to process the invitation, after calling XOnlineFriendsGameInvite, the title should call XOnlineTaskContinue with the handle returned by XOnlineFriendsStartup.

The XOnlineFriendsGameInvite function takes as a parameter the session ID of the online-game session to which the receiving users are invited. This session ID is available from the title that is hosting the session. That title obtains the session ID by calling XOnlineMatchSessionCreate, followed by XOnlineMatchSessionGetInfo. For more information about obtaining session IDs, see the reference pages for these functions and the Matchmaking Overview.

Receiving Invitations

Online titles must regularly check for game invitations, even during gameplay, and notify the player when a new invitation has arrived. Display of the invite notification during gameplay is expected to be unobtrusive. Games should signal players to pause the game when convenient, at which point more complete notification information and further UI can be displayed.

The recipient of an invitation receives a notification of type XONLINE_NOTIFICATION_TYPE_GAMEINVITE in his or her notification queue. The title running on the Xbox console of the receiving user should periodically call the XOnlineGetNotification function for each of the users on that console to check for the receipt of such notifications. The title should then inform the user(s) that they have a pending invitation.

To determine the identity of the user who sent the invitation, the receiving title enumerates the Friends list for the receiving user. The user who sent the invitation is present in the receiving user's Friends list, with a Friend state of XONLINE_FRIENDSTATE_FLAG_RECEIVEDINVITE. (The receiving user is also present in the inviting user's Friends list, with a Friend state of XONLINE_FRIENDSTATE_FLAG_SENTINVITE.) The Friend state is stored in the Friendstate field of the XONLINE_FRIEND structure.

The title on the receiving user's Xbox console should call XOnlineFriendsAnswerGameInvite with an appropriate value from XONLINE_GAMEINVITE_ANSWER_TYPE. In order to process the response to the request, after calling XOnlineFriendsAnswerGameInvite, the title should call XOnlineTaskContinue with the handle returned by XOnlineFriendsStartup.

If the receiving user accepts the invitation, the title should respond with XONLINE_GAMEINVITE_YES. The inviting user receives a notification of XONLINE_NOTIFICATION_TYPE_GAMEINVITEANSWER, and the receiving user's Friendstate contains the value XONLINE_FRIENDSTATE_FLAG_INVITEACCEPTED.

If the user chooses to decline the invitation, the response should be XONLINE_GAMEINVITE_NO. The inviting user receives a notification of XONLINE_NOTIFICATION_TYPE_GAMEINVITEANSWER, and the receiving user's Friendstate contains the value XONLINE_FRIENDSTATE_FLAG_INVITEREJECTED.

The receiving user also has the option to decline the invitation and remove the inviting user from his or her Friends list. In this case, the title should use the value XONLINE_GAMEINVITE_REMOVE in the call to XOnlineFriendsAnswerRequest. The inviting user receives a notification of XONLINE_NOTIFICATION_TYPE_FRIENDREQUEST. However, the title should not display this notification to the inviting user. Each user should siliently disappear from the other's Friend list.

The table below summarizes the notification and Friendstate values for the different game invitation scenarios.

Receiving User's Action Notification Received by Inviting User Receiving user's Friendstate in inviting user's Friends list
Receiving user accepts XONLINE_NOTIFICATION_TYPE_GAMEINVITEANSWER XONLINE_FRIENDSTATE_FLAG_INVITEACCEPTED
Receiving user rejects XONLINE_NOTIFICATION_TYPE_GAMEINVITEANSWER XONLINE_FRIENDSTATE_FLAG_INVITEREJECTED
Removes inviting user XONLINE_NOTIFICATION_TYPE_FRIENDREQUEST Receiving user disappears from inviting user's Friends list

Revoking Invitations

If a user has not enumerated their Friends list since receiving an invitation, the inviting user has the option of revoking the invitation. To revoke the invitation, the title calls XOnlineFriendsRevokeGameInvite. The task returned by the XOnlineFriendsEnumerate function must be active at the time the title calls XOnlineFriendsRevokeGameInvite. After calling XOnlineFriendsRevokeGameInvite, the title calls XOnlineTaskContinue with the handle returned by XOnlineFriendsStartup to process the revocation.

Setting the sessionID parameter to zero in a call to XOnlineNotificationSetState automatically expires all previously sent invitations, regardless of whether they have been enumerated by the receiving user.

Connecting to the Xbox Live Game Session

Once a user has accepted an invitation to join an Xbox Live game session, the title must connect to the Xbox console hosting the session. Follow these steps to establish the connection:

  1. Obtain the session ID of the online-game session from the sessionID member of the XONLINE_FRIEND structure of the user that sent the invitation.
  2. Call XOnlineMatchSessionFindFromID with this session ID to obtain an XONLINE_MATCH_SEARCHRESULT structure for the session.
  3. To verify that the session still has available private slots, check the value of the dwPrivateAvailable member of the XONLINE_MATCH_SEARCHRESULT structure.
  4. To register a secure-networking session, call the XNetRegisterKey function with the SessionID and the key exchange key from the XONLINE_MATCH_SEARCHRESULT structure.
  5. To obtain the IN_ADDR of the Xbox console that is hosting the session, call the XNetXnAddrToInAddr function with the SessionID and the Host Address from the XONLINE_MATCH_SEARCHRESULT structure.
  6. Lastly, send a join request to the hosting Xbox console. For more information about join requests, see Xbox Secure Network Library.

Cross-Title Invitations

It is possible that the user who receives the invitation is not playing the same title as the user that sent the invitation. The title on the receiving user's Xbox console can determine whether the title is different by calling XOnlineTitleIdIsSameTitle. The title can call XOnlineTitleIdIsSameTitle with the value of the titleID member of the XONLINE_FRIEND structure of the user that sent the invitation. If the title is different, the currently-running title should prompt the user to insert the disc for the new title. The currently-running title can retrieve the name of the new title by calling XOnlineFriendsGetTitleName with the title ID. The currently-running title should shut itself down, and prepare for the Xbox console to reboot.

It is also possible for the user to join a game session from a different title if the title is being played by a member of the user's Friends list. To do so, the currently-running title should call XOnlineFriendsJoinGame passing in the address of the XONLINE_FRIEND structure for the friend playing the new title. The current title should then prompt the user for the new title disc as though it was accepting a cross-title invitation

When the new title boots up, it should call XOnlineFriendsGetAcceptedGameInvite to check if an invitation to an online-game session is pending. All online-enabled titles should call this function when they first boot, even before calling XOnlineLogon or XOnlineFriendsStartup.

If there is no invitation or join pending, XOnlineFriendsGetAcceptedGameInvite returns S_FALSE. If a game invitation is pending, XOnlineFriendsGetAcceptedGameInvite returns an XONLINE_ACCEPTED_GAMEINVITE structure for the game invitation. This structure contains an XONLINE_FRIEND structure for the user that sent the invitation. It also contains an array of XUID structures for the users who accepted the invitation. As described earlier in this section, the title can use the information in these structures to establish a connection to the Xbox that is hosting the online-game session.

Notifications and State Information

The Notifications service is used to notify titles of real-time events. The list of possible events is currently restricted to Friend requests and game invitations, but the service may be extended in the future to support a wider variety of notifications. The notification system is designed primarily for Service-to-Xbox communication. It is not expected to support any form of real-time messaging.

Notifications are an automatic part of the underlying network traffic between the Xbox secure network stack and the Xbox Live service. On the client Xbox, incoming notifications are queued by the low-level network stack, and are retrieved by the title code with the XOnlineNotification family of functions.

None of the notification functions require an XOnlineFriendsStartup task or XOnlineFriendsEnumerate task to be active.

Function Description Memory Use
XOnlineGetNotification Access notification information. No
XOnlineNotificationSetState Sets presence information for a specified user. No

Notifications

The notifications can be interpreted by the title by means of the XOnlineGetNotification API, which allows Boolean conditions to be checked. This check is fast, does not generate any network traffic, and is only dependent on the XOnlineLogon task handle to be open. Hence, it is practical to use during gameplay.

The XONLINE_NOTIFICATION_FRIEND_REQUEST notification type returns true if any new Friend requests have been received from other players since either: 1) the last call to XOnlineLogon, or 2) the XOnlineFriendsEnumerate task was last closed. Titles may use this to detect the first new Friends request received during gameplay, or to indicate a new Friends request is present when displaying a menu.

The XONLINE_NOTIFICATION_GAME_INVITE notification type returns true if any new Friend requests have been received from other players since either: 1) the last call to XOnlineLogon, or 2) the XOnlineFriendsEnumerate task was last closed. Titles may use this to detect the first new game invitation received during gameplay, or to indicate a game invitation is present when displaying a menu.

The XONLINE_NOTIFICATION_NEW_GAME_INVITE notification type returns true if any new game invitation has been received from a friend since the last time this function was called. Titles may use this to detect each invitation received throughout gameplay, so the player can be notified by means of a glyph.

Currently, all notifications are only actionable by means of enumerating the Friends list.

Presence Information

Player state can be set by means of the XOnlineNotificationSetState function. This function does not immediately create any network activity, so it is practical to call during gameplay. The information specified by this function is communicated through the online heartbeat which is processed by the task created by XOnlineLogon. Redundant and rapid changes are consolidated throughout the system. Because of this, state information can take several seconds to propagate to a player's friends, and should not be used if more current information is available by means of the current game session. State information is split into three pieces: state flags, session ID, and state data.

State flags communicate core information about a player's state across all games by means of the Friends list.

The XONLINE_FRIENDSTATE_FLAG_ONLINE state flag indicates if the user should appear online to their Friends. If this flag is cleared, the player will appear offline to each of his or her Friends. The player will appear offline to each of his or her Friends unless the player: 1) sends an invitation to his or her Friends, or 2) accepts an invitation from his or her Friends. Titles may have a player appear offline at either the player's request or if the player is busy with non-game activity (such as downloading new content).

The XONLINE_FRIENDSTATE_FLAG_PLAYING state flag indicates if the player is in an active game session (as opposed to being in a lobby or navigating menus, and so on). Specific interpretation of this state flag is left up to individual titles.

The XONLINE_FRIENDSTATE_FLAG_VOICE state flag indicates whether the player's voice peripheral active. This state flag must be clear if: 1) the peripheral is inactive, or 2) the player's voice has been suspended by policy.

The XONLINE_FRIENDSTATE_FLAG_JOINABLE state flag indicates the player's session may be joined by his or her Friends without needing an invitation. To avoid lots of Friends racing for the same slot, it is recommended that titles should set this state flag only if there are a large number of open slots for a game.

The Session ID indicates which matchmaking session the player is currently in. If the player is not in a session, it must be zeroed out. Session ID must be updated when the player enters, creates, re-creates, or exits a game. It is used by the player's Friends to determine if game invitations are still current. Furthermore, it is used in combination with the XONLINE_FRIENDSTATE_FLAG_JOINABLE state flag to communicate which matchmaking session to join.

Finally, the pStateData field is 8 bytes that titles may use to encode title-specific data. Examples include current score, time remaining in game, character status (for example, surrounded by aliens and about to die), and character location.
