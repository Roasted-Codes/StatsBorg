Offering Packages

Offerings are built as "packages" containing offering information, content, and content signatures. Offerings are built using the BuildOffer tool.

Building Offering Packages

The following components are used when building an offering package:

  • A content directory — Contains all content that will be included in the package. The directory may have subdirectories, which will be preserved when the content is installed.
  • A content.xbx or subscription.xbx file — An offering metadata file which contains important information, supplied by the publisher, about the offering (such as name, price, language, unique identifier, and so on).
  • A contentimage.xbx file — An image file that is used to display an image for the content when it is viewed within the Xbox Dashboard.
  • Details and enumeration "blob" files — Files that contain the title-specific binary data used for the details and enumeration blobs. Two files must be provided for each language supported. The files are in the format details.xxx.bin and enum.xxx.bin, where xxx is a three-letter country code.

Place all content to be included in the package in the content directory. The root directory of the content will map to the content root directory on the Xbox when the content is later installed. After installation, the content's root directory on the Xbox hard disk can be determined by calling XGetContentInstallLocationFromIDs. The directory structure of the content directory is preserved when the package is installed on the hard disk.

The offering metadata file (content.xbx or subscription.xbx) and the details and enumeration blob data files (enum.xxx.bin and details.xxx.bin) do not need to be included in the content directory; they are referenced and incorporated into the content package when built and do not need to be installed as part of the content.

After the above components have been prepared and placed in their correct locations, run BuildOffer. BuildOffer should generally be run from the content root directory (or launched such that the content root directory is the current working directory).

The simple example below creates a package named test.xcp that includes all files in the current directory and its subdirectories:

BuildOffer -r N ..\test.xcp ..\content.xbx *.*

The location of the content.xbx file is specified in the BuildOffer command line, and the location of the 'blob' data files is specified within the content.xbx file itself. In this example, the offering metadata file (content.xbx) and the details and enumeration blob data files (enum.eng.bin, details.eng.bin, enum.jpn.bin, and details.jpn.bin) are located in the directory one level above the content root.

Note    If the content is offered on multiple continents, you must specify the -S option when invoking the BuildOffer tool. This option instructs BuildOffer to create multiple packages, one for each continent. The name of each package file will have a prefix that identifies which continent that package is for: NA for North America, EMEA for Europe, or JA for Japan. For more information, see the descriptions for Offering Identifier and Region Information in the Offering Metadata section below.

As part of the packaging process, BuildOffer produces a contentmeta.xbx file in the content root directory. This content metadata file is a signature file that contains content signatures for each file included in the content. This file needs to be installed when the content is installed, so it should either be explicitly specified in the content file list, or allowed to be picked up automatically with the recursive inclusion (-r) BuildOffer command-line switch.

The contentimage.xbx file is an image file, in the same format as a save game image file (SAVEIMAGE.XBX, 64x64 DXT1 formatted image), that is used to display an image for the content when it is viewed within the Xbox Dashboard. Additionally, a contentimage.xbx file can be placed at the root of the title's persistent data region (T:), to be used as the default content image file for that title. The dashboard will use the content's specific image file, if found; otherwise, the default title content image file will be used.

Offering Metadata (content.xbx or subscription.xbx)

The content.xbx and subscription.xbx files are offering metadata files that contain information supplied by the publisher concerning the offering. Values such as the name, price, language, and unique identifier of the offering are specified in this file. Example offering metadata files (content.xbx and subscription.xbx) are installed as part of the Xbox SDK, in the <XDKRoot>\xbox\bin directory.

Each section of the content.xbx file is described below, along with an example entry:

[Required] Version of this template (2.1):

Version=0x00020001

[Required] The type of offering (Content or Subscription):

OfferingType=Content

[Required] Whether the offering is a premium (requires purchase) offering (Yes or No):

Premium=No

[Subscription offerings only. Required] Offer Frequency. Valid values are: Onetime, Monthly, Quarterly, Biannually, or Annually:

OfferFrequency=Monthly

[Subscription offerings only. Required if Premium=Yes] Specifies whether the offer can be purchased more than once:

PurchaseOnceOnly=Yes

[Subscription offerings only. Required] Specifies the duration of the offer:

OfferDuration=12

[Subscription offerings only. Required] Specifies the free period for the offer, if any:

OfferFreePeriod=0

[Required] 32-bit Hexadecimal Title Identifier for which this content is created (substitute the title's actual identifier):

TitleID=0xffffffff

[Required] 64-bit Hexadecimal Offering Identifier assigned to this content package. The first 32 bits must match the title identifier. The next 32 bits specify:

  • A reserved bit that should be zero (1-bit).
  • The continent on which the content is offered (3-bits). Leave these bits zero. These bits are set by the BuildOffer tool.
  • Identifier determined by the publisher that is unique to this offer (28-bits)

The layout of the Offering Identifier is as follows:

<TID, 32-bits><Reserved, 1-bit><Continent, 3-bits><Publisher OID, 28-bits> 

OfferingID=0xffffffff00000000

Note    If the content is offered on multiple continents, the 3-bits that specify the continent must be zero, and you must specify the -S option when invoking the BuildOffer tool. The -S option instructs BuildOffer to create multiple packages, one for each continent. BuildOffer sets the continent bits to the correct value for each package. The name of each package file will have a prefix that identifies which continent that package is for: NA for North America, EMEA for Europe, or JA for Japan. For more information, see the description of Region Information below.

[Required] Publisher-defined bitflags to describe the offering. This can be any 32-bit number as desired by the publisher. This value is returned as the dwBitFlags value of the XONLINEOFFERING_INFO structure, and is used to filter content during enumeration:

BitFlags=0xffffffff

[Required] Date this offer becomes valid:

ActivationDate=3/1/2002

[Required] Date, if any, this offer expires:

EndDate=12/31/2999

[Required] Default display name for this package. This string will be used when none of the localized strings specified in the localized section below match the current language:

DefaultDisplayName="Sample Content Package"

[Content offerings only. Optional] The following set of properties specify localized display names used by the Dashboard to identify this offer based on the current language settings on the Xbox. The examples below contain localized display names for English (EN), Japanese (JA), and French (FR). If the Xbox happens to be set to a language other than these three, the string specified by DefaultDisplayName (above) will be used as default. Although DashDisplayName is optional, localized descriptions are highly recommended for readability and best user experience.

The following are the set of language codes supported by the Xbox.

Value Description
EN English
JA Japanese
DE German
FR French
ES Spanish
IT Italian
KO Korean
TW Traditional Chinese
BR Brazilian

Note that there is a counter after DashDisplayName, which must start from zero (0). That is, the first language in the list must be DashDisplayName.0, followed by .1, .2, and so on. The language codes can be listed in any order (for example, TW followed by EN is valid), but the counter must not leave any gaps (that is, 0, 1, 3 will cause anything after 1 to be ignored). This file is in UNICODE format so each localized string can be entered in its native code page. For example, glyphs may be used for JA, KO, and TW:

DashDisplayName.0=EN,"English Dash Display Name for this offer"
DashDisplayName.1=JA,"Japanese Dash Display Name for this offer"
DashDisplayName.2=FR,"French Dash Display Name for this offer"

[Required] Administrative friendly name. This is the administrative friendly name for this offer. This should be in English and is the name of the package known to Xbox Operations staff when referring to this offer:

AdminFriendlyName="Sample Content Package Friendly Name"

[Required] Region information for the regions in which this content is offered. If this content is offered in multiple regions, then multiple entries are required. The valid region values are:

    Value Region Continent
    5 Austria Europe (EMEA)
    8 Belgium Europe (EMEA)
    16 Canada North America (NA)
    25 Denmark Europe (EMEA)
    24 Germany Europe (EMEA)
    32 Finland Europe (EMEA)
    34 France Europe (EMEA)
    35 Great Britain Europe (EMEA)
    50 Italy Europe (EMEA)
    44 Ireland Europe (EMEA)
    74 The Netherlands Europe (EMEA)
    53 Japan Japan (JA)
    31 Spain Europe (EMEA)
    90 Sweden Europe (EMEA)
    18 Switzerland Europe (EMEA)
    103 USA North America (NA)

Note that the region values match the country values defined in xonline.h (XONLINE_COUNTRY_), not the region values returned by XGetGameRegion.

Note    If the content is offered on multiple continents, you must specify the -S option when invoking the BuildOffer tool. This option instructs BuildOffer to create multiple packages, one for each continent. The name of each package file will have a prefix that identifies which continent that package is for: NA for North America, EMEA for Europe, or JA for Japan. For example, if you specify USA (103), Canada (16), and Great Britain (35) as the regions in which the content will be offered, then you should specify the -S option in BuildOffer because these regions imply two continents: North America (NA) and Europe (EMEA). In this case, BuildOffer will create two package files. One package file will have a prefix of "NA" and the other will have a prefix of "EMEA".

The format of the region information is as follows:

    <Country ID>,<price whole>,<price fractional>,<tax type>

For free content, the whole and fractional price must also be zero. Leading and trailing whitespaces are not allowed before and after arguments. Tax type must be set to 'NotApplicable'.

If the content is for fee, then the whole and fractional prices must be set to the appropriate values. The correct tax type must also be specified. Valid values for tax type are: TaxNotIncluded, TaxIncluded, GSTIncluded, VATIncluded, or TaxNotApplicable.

Contact your account manager if you need more information about setting up offering regions.

The following example shows the content being offered in two regions:

OfferRegions.0=103,0,0,TaxNotIncluded
OfferRegions.1=16,0,0,TaxNotIncluded

Note that a single offering cannot be made available on multiple continents. For example, to make an offering available in both Japan and America, two separate offerings (with different identifiers) would need to be created. In this case, each offering package and metadata must be identical except for the offering identifier, regions offered, and price (if the price differs). It is permissible to make a single offering available in multiple regions in the same continent (for example, to Canada and the United States).

[Content offerings only. Optional] Specifies whether the content package should be displayed by the Dashboard. By default, a package is displayed by the Dashboard for management (for example, remove) purposes. This is the behavior if the DisplayInDash entry is omitted. However, if this entry is present and its value is set to 0, then the package will not be displayed by the Dashboard for management purposes. Titles, however, can still enumerate the package and manipulate it (for example, to remove it).

DisplayInDash=Yes

[Content offerings only. Optional] Download location URL for the package, in the following format:

    "<IP Address>[:<Port>]/<URL of package>"

(The extension of the package should be .xcp.)

This is for publishers who want to host their own download servers for privacy reasons. If this is not specified, the package will default to an Xbox-owned content download server on PartnerNet. Note that all content hosted on PartnerNet may be downloaded and viewed by other partners. You should consider hosting anything confidential on your own servers, or specify a SecurityKey (see below).

To use this feature, uncomment the line in the content.xbx file and substitute the desired URL:

; OfferLocations="1.2.3.4:3074/content/content.xcp"

[Content offerings only. Optional] Publisher security key. Because packages placed on PartnerNet are available to other publishers with PartnerNet access, a package that contains sensitive content can potentially be downloaded and viewed by other publishers. So to prevent others from seeing your unpublished content, you can specify an extra security key that provides additional security for your unpublished content. Simply uncomment the SecurityKey line in the content.xbx file and substitute the desired security key. This is a DEVKIT only feature. See XOnlineContentSetSecurityKey for more information:

; SecurityKey=0x00000000000000000000000000000000

[Required] Description sections for each description index. Description indices can be designed to mean anything and are up to the publisher to define and maintain for that specific title. Description indices may be any 32-bit number and the publisher may define as many description indices as required and they don't have to be sequential. Typically, description indices are used for localization and each index corresponds to a language or a region.

This section contains localizable data to description index 0.

[0]

[Required] Enumeration blob returned by XOnlineOfferingEnumerate for this content:

EnumerationData=c:\content\data\enum.0.bin

[Required] Details blob returned by XOnlineOfferingDetails for this content:

DetailsData=c:\content\data\details.0.bin

This section contains localizable data to description index 2. Note that it is permissible to leave gaps for description indices.

[2]
EnumerationData=c:\content\data\enum.2.bin
DetailsData=c:\content\data\details.2.bin

Content Metadata (contentmeta.xbx) and Signatures

The content signature file (contentmeta.xbx) produced by BuildOffer contains content signatures over each content file included in the offering. Before downloaded content is used by a title, the title must verify that the content is valid and has not been tampered with. This is done using the XLoadContentSignatures and XLocateSignatureByName functions to first open the content signature data, and then to locate the content signature for a particular content file. XCalculateContentSignature is then used to calculate a signature over the content file and compare the calculated signature with the known good extracted signature. If the two signatures match, the content is verified as being valid and unmodified. If the two signatures do not match, the title should not use the content.

The content signature file is similar to the signature files created by the XBldSigs tool, but is used for online offerings rather than offline offerings. Each signature is a named, automatic signature over a file included in the offering. Literal signatures (from either file lists or signature DLLs) cannot be specified. For more information about content signatures, see Content Signatures.
