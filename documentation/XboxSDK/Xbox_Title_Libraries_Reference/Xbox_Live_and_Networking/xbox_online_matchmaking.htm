Matchmaking Overview

This documentation provides an overview of the Xbox Live Matchmaking API. The overview covers the functions that make up the API, the purpose of each function, and how the functions should be called. The Xbox SDK Matchmaking sample demonstrates the use of these functions to provide simple matchmaking functionality.

The Matchmaking API exposes the functionalilty of the Xbox Live Matchmaking Service. Game titles use the Matchmaking Service to exchange information about currently running, network-enabled games ("online-game sessions"). An Xbox title that is hosting an online-game session uses the API to provide the service with information about its session. Other Xbox systems can then use the API to query the service to find appropriate online-game sessions to join. Both the host Xbox and the querying Xbox use Attributes to describe online-game sessions.

Attributes

Matchmaking attributes describe the characteristics of the available online-game sessions. Design a set of attributes for your title that enable other querying Xbox titles to distinguish between different types and levels of online-game sessions. Also, design a set of attributes to select the session(s) that will provide the best experience for the players on the querying Xbox console. The following are examples of attributes:

Examples of attributes:

  • Game type (for example, Deathmatch vs. Capture the Flag)
  • Required skill level to be in this game
  • Average skill of players in this session
  • Map or Level
  • Voice Peripheral required
  • Voice Peripheral turned off
  • Language spoken

Attributes pertain to the online-game session as a whole. An attribute does not contain player-specific information. If you want Matchmaking queries to contain data specific to the current players in the online-game session, aggregate the player data into a single attribute. For example, instead of storing the current world ranking of each player in an online-game session, store the highest or average world rank of all players as a single session attribute. If non-combinable player attributes are truly needed, titles can pull them from the individual game hosts after the title has received the Matchmaking query results.

Attributes are represented by the XONLINE_ATTRIBUTE structure.

Each attribute is identified by a unique DWORD value. In addition to uniquely identifying the attribute, this value also specifies the type of attribute, and whether the attribute is global or title-specific. This value is stored in the dwAttributeID member of the XONLINE_ATTRIBUTE structure.

Currently, there are no global attributes that are shared across all titles.

An attribute has one of three possible data types: Integer, String, or Binary Object (BLOB). An integer occupies eight bytes ( ULONGLONG). A string is Unicode encoded, is NULL terminated, and has a maximum size (including the NULL character) of 400 wide characters (800 bytes). A binary object is not NULL terminated; the dwLength member of the XONLINE_ATTRIBUTE structure contains the length of the binary object. A binary object has a maximum size of 800 bytes.

Public and Private Slots

In addition to attribute values, the number of players an online-game session can accommodate also affects the Matchmaking process.

An online-game session has two types of openings or "slots": public and private. The title that is hosting the session determines how many of each type of slot are available. A title may choose to have only public slots, only private slots, or a combination of both.

The game title needs to update the Matchmaking service with the number of slots of each type that are currently occupied in the session, and the number of slots of each type that are currently available for the session. Use the XOnlineMatchSessionCreate and XOnlineMatchSessionUpdate functions to provide the service with this information.

A player who finds and joins a session through a matchmaking query occupies a public slot. A matchmaking query returns only those online-game sessions where the number of public slots is greater than or equal to the number of authenticated users and guests on the querying Xbox console.

A player who joins a session at the invitation of a player already in the session occupies a private slot. For private online-game sessions (sessions that have only private slots), it is still necessary to call the XOnlineMatchSessionCreate function to obtain a session ID with which to invite other players. For more information on Invitation, see Friends Overview.

Asynchronous Xbox Live Task Architecture

The Xbox Live Matchmaking functions use an asynchronous online task architecture. In this architecture, an online task is not complete when the function returns. Instead, the function returns immediately with a task handle. Work is performed on the online task whenever the XOnlineTaskContinue function is called with this task handle. The return value of XOnlineTaskContinue indicates whether the task is complete. For more information about the asynchronous Xbox Live task architecture, see XOnlineTaskContinue.

Advertising Games to the Matchmaking Service

In order to manage information about its online-game session (which is stored in the Matchmaking service), the hosting Xbox uses the XOnlineMatchSession subfamily of functions.

Function Description Memory Use
XOnlineMatchSessionCreate Advertise a session. Yes - allocates memory for an Xbox Live task handle.
XOnlineMatchSessionGetInfo Retrieve information for an advertised session. No.
XOnlineMatchSessionUpdate Update a previously advertised session. No.
XOnlineMatchSessionDelete Stop advertising a session. No.

When a title is ready to make its session available to other players on a network, it calls XOnlineMatchSessionCreate to begin advertising the session on the Matchmaking service. This call provides the Matchmaking service with an initial set of attribute values and with initial player information. After the network task for this call is complete, the title calls XOnlineMatchSessionGetInfo with the task handle returned by XOnlineMatchSessionCreate. In addition to identifying the online task, the task handle also identifies the Xbox Live game session for the call to XOnlineMatchSessionGetInfo. XOnlineMatchSessionGetInfo then returns an ID for the session. In subsequent calls, this ID is used to identify the session. XOnlineMatchSessionGetInfo also returns a key-exchange key. Use the session ID and the key-exchange key in a call to XNetRegisterKey to register a secure networking session.

The information associated with an online-game session changes over time. The title must periodically update the Matchmaking service with any changes in the session information. How often the title needs to update the server depends on: 1) the attributes used, and 2) the design of the game title itself. However, almost all game titles should update the server at the following times:

  • Creating a new game
  • Loading a Game Level
  • Changing Game Options (Game Type, for example)
  • Players leaving or joining the session (changes in player configuration affect any session attributes aggregated for the current set of players)

Use the XOnlineMatchSessionUpdate function to update online-game session information in the Matchmaking service. The set of parameters for XOnlineMatchSessionUpdate is similar to that for XOnlineMatchSessionCreate. However, XOnlineMatchSessionUpdate also takes the session ID returned from XOnlineMatchSessionGetInfo. In addition, set the fChanged member of XONLINE_ATTRIBUTE to TRUE for attributes that are new or have changed since the call to XOnlineMatchSessionCreate.

The XOnlineMatchSessionDelete function stops advertising the online-game session on the Matchmaking service (in other words, the function removes any information about the session from the service). Call the XOnlineMatchSessionDelete function when the title no longer needs the Matchmaking service. This could occur when either: 1) the online-game session is finished, or 2) the title is no longer accepting new players for the session. The choice between the two preceding trigger events for calling XOnlineMatchSessionDelete is left to the discretion of the title.

Note that for certain games (join-in-progress games, for example), the title could continue to advertise the session even though the session is full. That way, if a player leaves the game (creating an opening for a new player), the title could update the session to reflect this fact by calling XOnlineMatchSessionUpdate.

Querying the Matchmaking Service

To obtain information about available online-game sessions, use the XOnlineMatchSearch subfamily of functions.

Function Description Memory Use
XOnlineMatchSearch Query for available online-game sessions. Yes - allocates memory for an online task handle.
XOnlineMatchSearchGetResults Retrieve query results. No.
XOnlineMatchSearchParse Parse the attributes returned with query results. No.
XOnlineMatchSessionFindFromID Locate an online-game session directly from the session ID. No.

The Matchmaking API provides two ways to obtain online-game session information: 1) using the XOnlineMatchSearch function to perform a query against the Matchmaking Service's online-game session database (where the query returns a set of online-game sessions matching the criteria specified in the query), and 2) calling the XOnlineMatchSessionFindFromID function with the session ID to obtain information about that particular online-game session.

Queries are based on attribute selection. All queries performed by the title have a (DWORD) index that maps to a stored procedure on the Matchmaking Service. The title supplies attribute values as parameters to the stored procedure. These attribute values depend on game-session selection algorithms (which may incorporate input from players) used by the title. The stored procedure uses these parameters to provide the desired result set of current online-game sessions.

Matchmaking cannot be used without at least one matchmaking query registered on the PartnerNet Matchmaking service for your title. If you want to incorporate Matchmaking into your game, work closely with Xbox Developer Support (xboxds@xbox.com) to create the matchmaking query (a database stored procedure) and have it installed on the service. Microsoft provides the Matchsim tool to assist you in developing stored procedures and client-side queries.

The Matchmaking sample that ships in the XDK uses a custom stored procedure to do matchmaking queries. The index of the stored procedure is passed to the XOnlineMatchSearch function. This stored procedure works only with the Matchmaking sample. Stored procedures are always uniquely associated with title IDs. This means that copying and pasting the Matchmaking code into your game will not work. The code is shown only as an example of how to use the APIs. Contact xboxds@xbox.com to set up custom queries for your game.

XOnlineMatchSearch and XOnlineMatchSessionFindFromID each return an online task handle. When XOnlineTaskContinue indicates that the online task is complete, pass this task handle to XOnlineMatchSearchGetResults to obtain the information for the online-game sessions.

The XOnlineMatchSearchGetResults function returns an array of XONLINE_MATCH_SEARCHRESULT structures. Each structure contains information for a particular online-game session. If the search was specified using XOnlineMatchSearch, the number of structures returned is less than or equal to the value of the dwNumResults parameter to XOnlineMatchSearch. If the search was specified using XOnlineMatchSessionFindFromID, just one structure is returned.

Appended to each XONLINE_MATCH_SEARCHRESULT structure is a buffer that contains the online-game session's attribute information. These attributes are distinct from the attributes that were passed as parameters to the stored procedure in the call to XOnlineMatchSearch.

You can use the XOnlineMatchSearchParse function to parse the attribute information. The XOnlineMatchSearchParse function accepts: 1) a pointer to one of the XONLINE_MATCH_SEARCHRESULT structures, and 2) an array of XONLINE_ATTRIBUTE_SPEC structures. The structures in this array describe the order and type of the attributes returned in the result. The XOnlineMatchSearchParse function parses the attribute information and then copies it to the output buffer pointed to by the pQuerySession parameter. This output buffer should contain a structure defined to hold the attribute values; in turn, this structure should be defined to use one-byte packing: 

pragma pack(1)

For example, suppose the stored procedure returned the following three attributes:

  • Skill represented by an Integer value.
  • Game Level represented by a String value of not more than 20 wide characters (including a terminating NULL character).
  • Game Type represented by a Binary Object value of not more than 40 bytes.

The values for the XONLINE_ATTRIBUTE_SPEC structures would look like the following: 


spec[0].type=X_ATTRIBUTE_DATATYPE_INT;
       .length=0;

spec[1].type=X_ATTRIBUTE_DATATYPE_STR;
       .length=20*sizeof(WCHAR);

spec[2].type=X_ATTRIBUTE_DATATYPE_BLOB;
       .length=40;

The definition for the structure pointed to by the pQuerySession parameter would look like: 


pragma pack(1)
struct {
	ULONGLONG		qwSkill;
	WCHAR		wszLevel[ 20 ];
	WORD		dwTypeLen;
	BYTE		bType[ 40 ];
}

Persisting a Matchmaking Session Across Reboots Between Game Levels

Game titles sometimes need to reboot between game levels. Use the following procedure to re-establish the matchmaking session after such a reboot.

Game Host/Server

  1. At the end of the first level, pass the current Session ID (XNKID) to the next level by means of the command line or a persistent memory page.
  2. Reboot into the next level.
  3. Create a new matchmaking session that is similar to the original, but set the "PreviousSessionID" attribute to the value passed in on the command line. This attribute is used to link the "thread" of match sessions together for the clients to follow. When advertising the first level, either don't set this attribute or set it to all zeros.
  4. Wait for all of the previous players to rejoin; perhaps allowing new join-in-progress players as well. The title may not want to open up any "official" publicly-available matchmaking slots until all previous clients have had a reasonable opportunity to rejoin the session (30 seconds or so, depending on the reboot cycle time). Alternatively, for join-in-progress games, the title could open up the number of slots that were previously still available. For example, if the title was playing with only five out of eight possible players, they could immediately advertise three slots.

Game Client

  1. At the end of the first level, pass the current Session ID (XNKID) to next level by means of the command line or a persistent memory page.
  2. Reboot into the next level.
  3. Run a matchmaking query that searches for sessions that have a "PreviousSessionID " (the old XNKID) passed as the only query parameter. This query will return either zero or one record, depending on the relative reboot speeds of the client and the host/server.
  4. If the title gets a record back from step #3, attempt to "rejoin" the session. The game host will need to redistribute the XNADDRs for all peers to every participating Xbox because these XNADDRs will have all changed across the reboot.
  5. If the title does not get a record back, the title should wait 5 to 10 seconds, and then go back to step #3. After about four retries, titles should give up and assume the host died during the transition.

Host Migration

In the event of a host disconnection, Host Migration is recommended. Host Migration is the ability of a title's networking system to automatically promote a new Xbox console to session host, with little or no effect on game play, if the current host is reset or drops its connection.

The best practice recommendation for host migration is for the new host to advertise a new session. All consoles currently in the game would already have established connectivity. Any new consoles will use the key associated with the new session. The new host would need to remember both old and new keys. This all assumes a client/server topology rather than a peer-to-peer topology. In a peer-to-peer topology the new key would have to be sent to all the consoles in the session and connectivity reestablished with the new key.

For more information on Host Migration, see the Host Migration sample.

Network Proximity in Matchmaking Queries

The Xbox network proximity system uses many overlapping sources of data to build a mapping table from IP addresses to geographical areas. It then computes an area-to-area distance matrix. The result of looking up the host IP address and the querying Xbox's IP address in the distance matrix becomes the "Network Distance" value used by the matchmaking servers to sort sessions by distance.

The "rough proximity" system takes all of the proximity distance values and groups them into five 20-percent ranges. These ranges, plus the possibility of two Xboxes being in the same range, yield six possible values: 0–5. This value is used as specified in the query definition to sort sessions. The rough proximity system enables the title to do "fuzzy" matching and prioritization, taking into account proximity as well as other sorting criteria.

Rough proximity is used in the following pattern of sort operators: 


<Game-specific sort criteria A-1>
      ...
<Game-specific sort criteria A-n>
<Sort by rough proximity ascending>
<Game-specific sort criteria B-1>
      ...
<Game-specific sort criteria B-n>
<Sort by network proximity ascending>

Typically, games using rough proximity will not include any sort operators in group A above. Also, there should always be at least one sort criteria separating rough proximity from network proximity. Otherwise, the rough proximity sort operator should be removed.
