Stats Overview

This topic provides an overview of the Xbox Live Stats API. The overview covers the functions in the API, the purpose of each function, and how the functions should be called.

The Stats API exposes the functionality of the Xbox Live Stats Service. Game titles use the Stats Service to store and retrieve statistical information about players of that game title. The type of statistical information stored depends on the nature of the title and is defined by the title developer.

Stats are stored on network servers and are accessible to instances of the game title running on any Xbox console on the network. Therefore, the player's stats remain with the player regardless of which Xbox console he or she is using.

Stats collected by the title over a series of game sessions can be used to track the player's performance on the title over time. For example, the title might track the player's top five scores and the dates they were obtained, or the player's 10-day, 30-day, and 90-day moving averages.

The Stats Service is also able to provide ranking information about the performance of players of the title. That is, the service can compare a given player's performance to that of other players. Unlike traditional rankings in which a game title can only compare players that have played on the same console, the Stats Service is able to rank players who have played the title on any network-enabled Xbox console.

Different titles require different types of statistical information. The Stats Service supports storing any number of different statistical values per user. The title developer decides which stats to collect.

Xbox Live titles should consider how they will protect against Guests being used to artificially improve a player's measured performance. For example, in a deathmatch-style game, Guests could be used to improve a player's stats by pre-arranging kills with the Guest. Titles can protect against this type of cheating by adjusting their rules when Guests are present.

Leaderboards

A leaderboard (often referred to as a "scoreboard") is an ordered, title-specific listing of players and their stats. When a title stores a player's stats with the Stats Service, the title specifies a leaderboard with which to associate the stats. Each leaderboard is identified by a unique ID. Title developers should contact Xbox ATG to obtain unique IDs for their leaderboards.

Leaderboards are always exclusive to a particular title; they are never shared between titles.

The Stats Service supports multiple leaderboards per title. The current limit on the number of leaderboards per title is 20. A title may store the stats for a user on more than one leaderboard.

Rankings and Ratings

The listing of the players on a leaderboard is ordered. The position of the player in the listing is the player's rank on that leaderboard. The ranking of players on the leaderboard is maintained by the Stats Service. To determine a player's ranking relative to other players, the service uses the player's rating on that leaderboard. The rating is a signed 64-bit number that is calculated by the title. The Stats Service ranks players with higher ratings above players with lower ratings; the player in the top position on the leaderboard is the player with the highest rating, and has a ranking equal to one.

Because the Stats Service allows titles to have multiple leaderboards, a title is able to rank players in multiple ways. For example, a title might have daily, weekly, and monthly leaderboards. In this case, a given player might have the same rating, but a different rank on each of the leaderboards.

Stats Decay

The Xbox Live Stats Service provides a feature called Stats Decay. The Stats Decay feature removes a player from a given leaderboard if the player has not played the title within a given period of time. The amount of time before the removal occurs is specified in the leaderboard configuration information provided by the publisher using XLAST. The Stats Decay feature creates an incentive for successful players to continue to play in order to maintain their rankings.

Because of Stats Decay, it is important that a title write to the leaderboards on which a player has posted stats every time the player plays the title. The title should write to the leaderboard regardless of whether the player has achieved a higher rating than previously posted. The reason is that each leaderboard maintains a "last-write" time stamp for each user. When a title writes data for a player to a leaderboard, it refreshes the time stamp for that player on that leaderboard.

Writing any value to the leaderboard will refresh the time stamp. However, if the title is using the Game Clips feature, the title should not rewrite the player's rating. Updating the rating, even if the same value is used, will cause the service to remove the previous attachment associated with that rating and return S_CAN_UPLOAD so the title can upload a new attachment. To avoid updating the attachment, the title must write a statistic other than the rating.

Leaderboard Resets

Title developers can arrange with the Stats Service to have the leaderboards for a title reset periodically. When a leaderboard is reset, all data on the leaderboard is deleted. Title developers can arrange to have different leaderboards reset at different intervals. For example, a title might have three leaderboards associated with it, one of which could be reset daily, the second weekly, and the third monthly. The following reset intervals are supported:

  • Weekly
  • Biweekly (that is, every other week)
  • Monthly
  • Quarterly
  • Yearly
  • Never

The title itself is also able to reset the stats for a single user by calling the XOnlineStatReset function.

Stats

Game titles are allowed to collect and store whatever stats are appropriate for their game design. Each leaderboard is able to store up to 64 stats per user. The title does not have to prearrange with the Stats Service the number or type of the stats stored.

The following sections describe:

  • The data types that are allowed for stats.
  • The stats that are defined by the Stats Service.

Types of Stats

Statistical values must be numeric and belong to one of three possible data types. These types correspond to the values of the XONLINE_STAT_TYPE enumeration type:

Integer; 32-bits XONLINE_STAT_LONG
Big integer; 64-bits XONLINE_STAT_LONGLONG
Real number; 64-bits XONLINE_STAT_DOUBLE

The Stats API also provides the XONLINE_STAT_LPCWSTR data type for storing a user's title-specific nickname with his or her stats. This type should not be used for any other purpose.

Stats Defined by the Stats Service

In addition to the stats that the title defines, there are eight stats defined by the Stats Service itself.

Identifier Description Type
XONLINE_STAT_RANK Rank stat. Ranks have a type of XONLINE_STAT_LONG (32 bits).
XONLINE_STAT_RATING Rating stat. Ratings have a type of XONLINE_STAT_LONGLONG (64 bits).
XONLINE_STAT_NICKNAME User nickname. Titles should not use this stat. Additionally, it is XOnlineStatWriteEx prevents this stat from being specified. Nicknames have a type of XONLINE_STAT_LPCWSTR (variable-length string).
XONLINE_STAT_LEADERBOARD_SIZE Number of entries on the leaderboard. The leaderboard size has a type of XONLINE_STAT_LONG (32 bits).
XONLINE_STAT_ATTACHMENT_PATH Game Clip path. Attachment paths have a type of XONLINE_STAT_LPCWSTR (variable-length string).
XONLINE_STAT_ATTACHMENT_SIZE Size of Game Clip. The attachment size has a type of XONLINE_STAT_LONG (32 bits).
XONLINE_STAT_UNIT_ACTIVITY_COUNTER Increment-only stat used to prevent unit from expiring. Activity counters have a type of XONLINE_STAT_LONG (32 bits).
XONLINE_STAT_UNIT_LAST_ACTIVITY_DATE Read-only stat time unit was last used. Last activity dates have a type of XONLINE_STAT_LONGLONG (64 bits).

In the same way that a set of stats for a user is always associated with a particular leaderboard, the title-specific nickname for a user is also associated with a particular leaderboard. Because the nickname is handled as a stat, the nickname for a user on a leaderboard could change each time the user's stats are updated. Additionally, it is possible for a given user to be known by different nicknames on other leaderboards.

The Stats API

The following table summarizes the functions in the Stats API:

Function Description
XOnlineStatWrite Stores stats for one or more users on specified leaderboards.
XOnlineStatReset Resets (deletes) stats for a user on the specified leaderboard.
XOnlineStatRead Creates an asynchronous task to retrieve stats for one or more users from specified leaderboards.
XOnlineStatReadGetResult Obtains the results of the asynchronous task created by XOnlineStatRead.
XOnlineStatLeaderEnumerate Creates an asynchronous task to retrieve an ordered listing of users from a specified leaderboard.
XOnlineStatLeaderEnumerateGetResults Obtains the results of the asynchronous task created by XOnlineStatLeaderEnumerate.

Setting and Changing Stats

The XOnlineStatWrite and XOnlineStatReset functions change the information stored for specified users. Each of the functions takes as a parameter an XUID that specifies the user whose information should be changed. For security, the functions can modify information only for users that are logged on to the Stats Service from the local Xbox console.

The XOnlineStatWrite function is used to set stats for a specified group of users. The XOnlineStatWrite function takes an array of XONLINE_STAT_SPEC structures that specify the users for whom to set stats. The dwLeaderBoardID member of each XONLINE_STAT_SPEC structure specifies the leaderboard on which to set the stats. Because the leaderboard is specified on a per-user basis, a single call to XOnlineStatWrite can be used to set stats for users on multiple leaderboards. The pStats member of each XONLINE_STAT_SPEC structure points to an array of XONLINE_STAT structures that specify the stats to set and the values to which they should be set. Use the XONLINE_STAT structures to specify both title-defined stats and the service-defined stats rating and nickname, but not rank (see below). Because the stats are specified on a per-user basis, a different set of stats can be set for each user. For maximum performance, the title should set all the results from a game-session using a single call to the XOnlineStatWrite function.

Because the ranking of a user on a particular leaderboard is calculated by the Stats Service, a title is not able to set the ranking with XOnlineStatWrite. However, the title should set a rating stat (XONLINE_STAT.wID==XONLINE_STAT_RATING) because the Stats Service determines the ranking based on the rating. As a best practice, titles should store all the information that they use to calculate the rating as stats on the same leaderboard. This practice allows for the possibility of server-side calculation of ratings in the future.

If the title does not set a rating stat for the user, then the user's rating and rank are undefined. Although the user's stats can be retrieved using XOnlineStatRead, the user does not appear in leaderboard lists returned by XOnlineStatLeaderEnumerate. Also, the user is not counted in the number of users on the leaderboard (XONLINE_STAT.wID==XONLINE_STAT_LEADERBOARD_SIZE) reported by XOnlineStatLeaderEnumerate.

The XOnlineStatReset function deletes all stats data for a specified user on a specified leaderboard. As with XOnlineStatWrite, the specified user must be logged on to the Stats Service from the local Xbox console.

For testing on PartnerNet only, if the title specifies a value of zero for the qwUserID member of the xuid parameter to XOnlineStatReset, the Stats Service resets all user data on the leaderboard specified by the dwLeaderBoardId parameter. If both the qwUserID member and the dwLeaderBoardId parameter are zero, the Stats Service resets all user data on all leaderboards for that title. However, these special testing features do not work with the Stats Sample.

Retrieving Stats

The XOnlineStatRead and XOnlineStatReadGetResult functions retrieve stats for a specified group of users. The XOnlineStatLeaderEnumerate and XOnlineStatLeaderEnumerateGetResults functions retrieve an ordered contiguous listing of users from a specified leaderboard.

A title can retrieve a user's stats with these functions, even if that user is not logged on to the Stats Service.

As already stated, the XOnlineStatRead and XOnlineStatReadGetResult functions are used together to retreive stats for a specified group of users. The XOnlineStatRead function takes an array of XONLINE_STAT_SPEC structures that specify the users for whom to obtain stats. The dwLeaderBoardID member of each XONLINE_STAT_SPEC structure specifies the leaderboard from which to obtain the stats. Because the leaderboard is specified on a per- user basis, a single call to XOnlineStatRead can be used to obtain stats from users on multiple leaderboards. The pStats member of each XONLINE_STAT_SPEC structure points to an array of XONLINE_STAT structures that specify which stats to retrieve. Use the XONLINE_STAT structures to specify both title-defined stats and service-defined stats such as rating, rank, and nickname. Because the stats to retrieve are specified on a per-user basis, a different set of stats can be obtained for each user. XOnlineStatRead creates an asynchronous task to retrieve the stats from the Stats Service. When this task completes, call XOnlineStatReadGetResult to obtain the stats. The title must pass XOnlineStatReadGetResult the same set of XONLINE_STAT_SPEC and XONLINE_STAT structures that the title passed to XOnlineStatRead.

The XOnlineStatLeaderEnumerate and XOnlineStatLeaderEnumerateGetResults functions retrieve an ordered contiguous listing of users from a specified leaderboard. The listing is sorted by user ranking. Unlike the XOnineStatRead/GetResult functions, the XOnlineStatLeader functions retrieve users and their stats from only a single leaderboard per call. Also, the set of stats retrieved is the same for each user.

The XOnlineStatLeaderEnumerate function creates an asynchronous task to retrieve the leaderboard entries from the Stats Service. Pump this task by calling XOnlineTaskContinue with the task handle returned by XOnlineStatLeaderEnumerate. When XOnlineTaskContinue returns XONLINE_S_SUCCESS, call XOnlineStatLeaderEnumerateGetResults to obtain the results. Next, call XOnlineTaskClose on the task handle.

The dwPageSize parameter to XOnlineStatLeaderEnumerate specifies the number of users to retrieve. The title can specify which users to retrieve in one of two ways. The first is to specify the leaderboard entry at which to begin the enumeration. Specify the entry in the dwPageStart parameter to XOnlineStatLeaderEnumerate. For example, to start the enumeration at the top of the leaderboard, specify a value of one. A second way to specify the users to retrieve is to specify a user that should be located in the middle of the list. Specify the user by passing the XUID for the user as the pxuidPagePivot parameter. XOnlineStatLeaderEnumerate attempts to retrieve an equal number of users above and below the specified pivot user.

The XOnlineStatLeaderEnumerateGetResults function retrieves the leaderboard entries requested by XOnlineStatLeaderEnumerate. The pUsers parameter points to a buffer that receives an array of XONLINE_STAT_USER structures that correspond to the users retrieved. The pStats parameter points to a buffer that receives an array of XONLINE_STAT structures containing the stats for these users. The pdwReturnedResults parameter receives the number of users retrieved, and the pdwLeaderboardSize parameter receives the total number of users listed on the leaderboard.
