Authentication Overview

This document describes the functions available to title developers for the purpose of Xbox Live authentication to enable secure online gameplay. This functionality allows title developers to establish their own secure Xbox Live gaming infrastructure and user account system, worry-free. The functions described in this overview are used only to achieve authentication and to provide enough infrastructure for the titles to communicate securely with other Xbox Live services. The functions used to communicate with other Xbox Live services, such as Matchmaking and Content Download, are not discussed here.

Goals

The goal of the authentication functions is to provide game developers with an easy way to securely authenticate the user to the XBox Live service without the complexities of authentication and security. These functions were designed based on the following guidelines:

  • Simple and lightweight

    • In order to meet the needs of Xbox console developers, the primary goal is to provide simple and lightweight functions. Developers should not have to load large amounts of code or rely upon heavy abstractions in order to deal with online gameplay.

  • Provide maximum security

    • The main purpose of the Xbox Live authentication functions is to enable secure and trusted online gameplay, such that all Xbox Live identities can be trusted to be who they say they are. The functions also provide mutual authentication capability between Xbox consoles and Xbox Live services.

  • Enable title developers

    • After Xbox Live authentication has been established, we enable title developers to communicate with various Xbox Live services without worrying about any authentication details.

Overview

Before Xbox Live functionality can be carried out, a user account (an Xbox Live identity) must be created. User accounts are created and maintained using the Xbox Dashboard. As a result, no functions are exposed, and the title developer is not burdened with such tasks. Title developers should assume, for the purpose of this document, that at least one user account has been created and resides on one of the following: 1) the XBox console, or 2) an Xbox Memory Unit (MU) inserted into a controller that is connected to the XBox console.

The Xbox Live authentication functions enable a title to enumerate the Xbox Live users who have their information stored either on the local Xbox hard disk or on the Xbox MUs. The title can then prompt the users to find out which one of the stored users they are represented by, and optionally ask for the pass code if one is required. The title can then proceed to log on to the Xbox Live authentication system using the user identities just obtained. Titles must prevent multiple players on the same Xbox from using the same account; each player must have a valid User account or log in as a Guest of another player. After logging on, the title can then use the other functions to communicate with various Xbox Live services.

The underlying implementation of these authentication functions uses the Kerberos protocol to establish mutual trust between each authenticated Xbox game session and various Xbox Live services. Kerberos is a standard protocol widely deployed and used for authentication purposes. For more information on Kerberos and other standards, please visit http://www.ietf.org.

Authentication Functions

The following list contains the functions currently planned or implemented for Xbox Live authentication:

Function Description Memory Usage
XOnlineGetUsers Retrieves Xbox Live users stored on the local hard disk and Xbox MU. No
XOnlineLogon Logs on to the Xbox Live system on behalf of the specified users and obtains tickets for the specified services. Yes - Allocates enough memory to hold information for all logged on services.
XOnlineChangeLogonUsers Changes the set of local users logged onto the Xbox Live service. Yes - Allocates enough memory to hold information for all logged on services.
XOnlineTitleUpdate Downloads an updated version of this game title and reboots. Yes - but is freed after the Asynchronous task is completed.
XOnlineGetLogonUsers Retrieves a pointer to an array of users currently logged on. No
XOnlineGetServiceInfo Retrieves IP, and port of a logged-on Xbox Live service. No

Enumerating Local Xbox Live Users

A fixed set of Xbox Live users (XONLINE_MAX_STORED_ONLINE_USERS to be exact) can have their information stored on the local Xbox hard disk and all mounted Xbox Memory Units (MUs). The XOnlineGetUsers function is used to retrieve all Xbox Live users with information stored on the local Xbox hard disk and all mounted Xbox MUs.

Collecting User Input for the Pass Code

Users may enable a pass code on their Xbox Live account to prevent the account from being used by others. If a user enables the pass code, the title needs to collect this pass code from the player using the Xbox Controller. Each control on the controller corresponds to a byte value as specified by the XONLINE_PASSCODE_TYPE enumeration type. The length of the pass code is specified by the XONLINE_PASSCODE_LENGTH constant as defined in XOnline.h. The pass code must be exactly this length. The title collects the pass code from the user and compares it with the value in the passcode member of the XONLINE_USER structure for the user. The user should be allowed to log on only if there is a match between the pass code and this value. The comparison should be made only after all the bytes of the pass code have been collected from the user.

Log On to Xbox Live

The XOnlineLogon function is used to start an asynchronous logon process. This logon process involves communicating with the authentication server and using Kerberos to authenticate users and obtain tickets used for future communication with various requested Xbox Live services. The service identifiers returned by this routine can be used in various Xbox Live functions. XOnlineLogon can be called again, but all service handles from the previous logon call will be lost. Therefore, we recommend that Xbox Livetitles first collect all user information and service names ahead of time, and then only make one logon call. If the title supports adding or removing users midway through the game, then the title will need to call XOnlineChangeLogonUsers again with the new set of users. Titles should prompt new users for their optional pass code but not bother users who have already logged on.

XOnlineLogon and XOnlineChangeLogonUsersare NOT thread safe. Titles should avoid making a second call to either of these functions before the task for the first call has completed; otherwise, the function fails with an error of XONLINE_E_TASK_BUSY.

Getting the List of Users Currently Logged On

After calling XOnlineLogon, a title may wish to retrieve the list of logged-on players for their own use. Instead of having to store this data, titles can call XOnlineGetLogonUsers to achieve this.

There are no parameters to this call. The result of this call is a pointer to an array of XONLINE_USER structures that hold information about the players who are currently logged on. This array will always contain XONLINE_MAX_LOGON_USERS. The title must go through the individual data structures to determine whether there is a user logged on to a specific controller port. The array returned has the user's index in controller port order. That is, the index of the array, 0–3, corresponds to the controller port, 1–4, respectively. To identify whether a user is logged in, titles need to qwUserID. If this is zero, no user is logged on at this controller port. The function will return NULL when a logon has not been completed. After a logon is completed, it will always return an array, even if no users are logged on.

Test the hr member of the XONLINE_USER structure for each user to determine if there was a logon failure specific to that user. See the reference page for XONLINE_USER for a list of possible errors.

Test the dwUserFlags member of the XUID member of the XONLINE_USER structure to determine any limitations that have been placed on the user's account—for example, whether the user is not allowed to use voice chat or not allowed to choose custom nicknames.

Updating a Game Title

If the result code of the logon task (obtained with XOnlineLogonTaskGetResults) is XONLINE_E_LOGON_UPDATE_REQUIRED, the game title will need to perform an update before any Xbox Live gameplay can occur for this title. This update is performed by calling XOnlineTitleUpdate. XOnlineTitleUpdate will reboot the console, download the updated binaries for this game title onto the hard disk, and reboot back to the game title. Upon reboot, the title will be able to successfully logon from that point on (at least until the next update is released). Before calling XOnlineTitleUpdate, the game title should inform the player that an update is about to happen, and that the Xbox will reboot for the update. The player should have the option to NOT update at this time and possibly proceed with playing the game without any Xbox Live functionality.

Note    XOnlineTitleUpdate must occur only after a XONLINE_E_LOGON_UPDATE_REQUIRED. Titles should not directly call it without first calling XOnlineLogon, and then pumping the returned task until completion. Nor should titles call it if the logon task returns anything other than XONLINE_E_LOGON_UPDATE_REQUIRED.

Obtaining Service Info and Ticket

After a successful logon, the title can communicate with the various logged on services to perform the desired Xbox Live functionality. Titles may also directly access the IP address and port of the service.

To find out information about a service to which the title is logged on, use the XOnlineGetServiceInfo function.

Guests

Xbox Live titles must support Guests. Guest are users who have not yet subscribed to Xbox Live, but who are interested in exploring the service. Up to three Guests may play concurrently with one subscribed user. Not all Xbox Live features are available to Guests; Guest may not use the voice chat feature or purchase content.

Xbox Live titles should consider how they will accommodate the presence of Guests. For example, titles need to protect against Guests being used for cheating. In a deathmatch-style game, Guests could be used to improve a player's statistics artificially by prearranging kills with the Guest. Titles can protect against this type of cheating by adjusting their rules when Guests are present.
