Thread Stack Size

Each new thread receives its own stack space, consisting of a region of committed, reserved memory. By default, each thread uses 64 KB of memory. The thread's stack is allocated when the thread is created, and will not grow beyond that initial size. The end of the stack is protected by an uncommitted page entry. Overflowing the stack results in a stack overflow exception that cannot be caught by a title's exception handler. To specify a different default stack size, use the /STACK ImageBld flag, or the /STACK linker flag.

To increase the amount of stack space which is to be initially committed for a specific thread, specify the size in the dwStackSize parameter of the CreateThread function. This value is rounded to the nearest page and used to set the size of the committed memory for the stack. The call to CreateThread will fail if there is not enough memory to commit the number of bytes requested. A minimum of 12 KB will be committed for the thread's stack, even if a smaller size is specified.

Always use the default (or larger) stack size for threads that will be calling XTL functions. Using stack sizes smaller than 64 KB can cause XTL functions to crash or behave unexpectedly.

The stack is freed when the thread terminates.

Xbox threads run in kernel mode (ring 0). ISRs (interrupt service routines) generated by the system software run on the stack of the current thread, not on any system worker threads. Titles should ensure that there is always enough free thread stack space to service those calls (a minimum of 4 KB free on the stack at all times is recommended). Additionally, since ISRs rely on the thread's stack, the ESP register should not be used by a title for purposes other than pointing to the correct location in the stack, as that register will be used by the ISRs when they execute. Also note that although DPCs (delayed procedure calls) are run on the currently executing thread, they use their own stack rather than the stack of the thread that they run on.
