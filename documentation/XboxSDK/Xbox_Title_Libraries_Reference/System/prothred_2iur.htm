Multithreading Considerations

The recommended guideline is to use as few threads as possible, thereby minimizing context switching and the use of system resources — doing so can often improve performance. Multithreading has resource requirements and potential conflicts that need to be considered. The resource requirements are as follows:

  • The system consumes memory for the context information required by threads. Therefore, the number of threads that can be created is limited by available memory.
  • Keeping track of a large number of threads consumes significant processor time. If there are too many threads, most of them will not be able to make significant progress.

Providing shared access to resources can create conflicts. To avoid them, synchronize access to shared resources. This is true for system resources (such as input and output devices) and for resources accessed by multiple threads (such as global variables). Failure to synchronize access properly can lead to problems such as deadlock and race conditions. The XTL provides a set of synchronization objects and functions you can use to coordinate resource sharing among multiple threads. For more information about synchronization, see Synchronizing Execution of Multiple Threads. Reducing the number of threads also makes it easier and more effective to synchronize resources.

In many cases in the XTL, rather than using background worker threads, routines are exposed to titles that can be called to perform the necessary operations. The DoWork and DirectMusicDoWork methods in DirectPlay and DirectMusic (respectively) are examples of such functions. Titles can call these routines at appropriate places in their game loop (for example, while waiting on the GPU) in order to perform processing needed by those components.
