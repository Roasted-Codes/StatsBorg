SetThreadPriority

The SetThreadPriority function sets the priority value for the specified thread.

BOOL SetThreadPriority(
  HANDLE hThread, // handle to the thread
  int nPriority   // thread priority level
);

Parameters

hThread
[in] Handle to the thread whose priority value is to be set.
nPriority
[in] Specifies the priority value for the thread. This parameter can be one of the following values:
Priority Meaning
THREAD_PRIORITY_IDLE Indicates a base-priority level of 1.
THREAD_PRIORITY_LOWEST Indicates 2 points below normal priority.
THREAD_PRIORITY_BELOW_NORMAL Indicates 1 point below normal priority.
THREAD_PRIORITY_NORMAL Indicates a base-priority level of 8.
THREAD_PRIORITY_ABOVE_NORMAL Indicates 1 point above normal priority.
THREAD_PRIORITY_HIGHEST Indicates 2 points above normal priority.
THREAD_PRIORITY_TIME_CRITICAL Indicates a base-priority level of 15.

This parameter can also be -7, -6, -5, -4, -3, 3, 4, 5, or 6.

Return Values

If the function succeeds, the return value is nonzero.

If the function fails, the return value is zero. To get extended error information, call GetLastError.

Remarks

Every thread has a base priority level determined by the thread's priority value. The system uses the base priority level of all executable threads to determine which thread gets the next slice of CPU time. Threads are scheduled in a round-robin fashion at each priority level, and only when there are no executable threads at a higher level does scheduling of threads at a lower level take place.

The SetThreadPriority function enables setting the base priority level of a thread. For example, specifying THREAD_PRIORITY_HIGHEST in a call to SetThreadPriority sets the thread's base priority level to 10.

The system dynamically boosts a thread's base priority level when events occur that are important to the thread.

All threads initially start at THREAD_PRIORITY_NORMAL. Use the GetThreadPriority function to get the priority value of a thread.

Use thread priority values to differentiate the relative priorities of tasks. For example, a thread that handles input could have a higher priority level than a thread that performs intensive calculations for the CPU.

When manipulating priorities, be very careful to ensure that a high-priority thread does not consume all of the available CPU time. A thread with a base priority level above 11 interferes with the normal operation of the system software. Using THREAD_PRIORITY_TIME_CRITICAL may cause disk caches to not flush, and so on. In general, a high priority thread should spend most of its time in a wait state so that it doesn't interfere with other threads. The high priority thread completes the wait, quickly performs some operation, and then re-enters the wait state.
