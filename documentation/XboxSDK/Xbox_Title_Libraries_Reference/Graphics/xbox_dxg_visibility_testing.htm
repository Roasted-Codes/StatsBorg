Visibility Testing

Xbox provides a way to test how many pixels will be drawn on the screen for a particular set of geometry. This feature can be used to determine whether geometry should be drawn and what level of detail should be used.

A visibility test is started by calling IDirect3DDevice8::BeginVisibilityTest. This clears the pixel counter in the chip. An application then sends test geometry to the chip. This test geometry is typically a bounding box or other bounding volume that completely encloses the model being tested for visibility. The test geometry is typically much simpler than the geometry being tested for visibility. The application calls IDirect3DDevice8::EndVisibilityTest to indicate to the hardware that the test geometry has been sent.

To reduce the memory automatically allocated, keep index values as low as possible and reuse them where feasible.

The use of any indices within a 256 range, aligned to multiples of 256, will cause the allocation of a 4K buffer. Subsequent usage of any indices in the same 256 range will not cause any additional allocations. The memory will be reused.

For example, using indices 0, 1, and 2 allocates a single 4K buffer. These indices all fall within 0–255. If you use only index 300, this also allocates a single 4K buffer because it falls within 256–511 (a 256 range, aligned to a multiple of 256.) However, if IDirect3DDevice8::EndVisibilityTest is called using indices 0, 1, 2, and 300, the visibility testing allocates a total of 8K (4K for 0–255 and 4K for 256–511.)

A visibility test is pipelined along with the other geometry and render states being submitted to the chip. This makes it possible to queue up several visibility tests at one time. Each visibility test is assigned an integer identifier by the application. Direct3D allocates a number of results slots equal to or greater than the largest integer identifier specified. When the visibility test is completed by the hardware, it writes the number of pixels rendered for the test geometry into the slot corresponding to the integer ID for that test. The application can then retrieve the results by calling IDirect3DDevice8::GetVisibilityTestResult and passing the ID for the desired test.

It is up to the application to set the appropriate rendering state before beginning the visibility test. For example, if it is not desired to actually render the test geometry, color and Z writes should be disabled so that the contents of the frame buffer and Z-buffer are not changed.
